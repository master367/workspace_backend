## Проект: Internal Team Workspace Backend

## 1. Общая идея проекта

Нужно реализовать backend для внутреннего рабочего пространства команды.

Это **не SaaS** и **не multi-tenant система с организациями**.  
Проект предназначен для **одной внутренней команды/компании**, где сотрудники работают с клиентами, задачами и внутренними действиями.

Система должна позволять:

- управлять пользователями
- хранить клиентов
- работать с задачами
- разграничивать права по ролям
- фиксировать действия пользователей в audit logs

---

# 2. Основные сущности проекта

## Users
Пользователи системы.  
Создаются не через публичную регистрацию, а администратором.
## Clients
Клиенты компании.  
Содержат основную информацию: контакты, статус, источник и другие базовые данные.
## Tasks
Внутренние задачи команды.  
Отображаются как kanban-доска со статусами.
## Audit Logs
Журнал действий пользователей.  
Нужен для ответа на вопросы:
- кто создал запись
- кто изменил запись
- кто поменял статус
- кто взял задачу

---
# 3. Роли пользователей
На старте достаточно следующих ролей:
- `admin`
- `teamlead`
- `member`
## Предварительные правила
- `admin` может создавать пользователей
- `admin` может создавать клиентов
- `admin` и `teamlead` могут создавать задачи
- остальные ограничения можно вводить поэтапно

Роли не должны превращаться в сложную RBAC-систему.  
На данном этапе они нужны для:
- базовых прав доступа
- отображения ответственности
- аудита действий

# 5. Разделение ответственности
## Router

Отвечает за:
- HTTP endpoints
- request/response schemas
- получение зависимостей
- вызов service

Router **не должен**:
- писать SQL
- содержать бизнес-логику
- напрямую работать с БД кроме передачи зависимостей

## Service

Отвечает за:
- бизнес-логику
- правила доступа
- orchestration между repository и другими частями
- вызовы audit log при важных действиях

## Repository
Отвечает за:
- чтение и запись в БД
- SQLAlchemy queries
- CRUD операции
- поиск, фильтрацию, списки

- FastAPI

- PostgreSQL
- Docker Compose
- JWT authentication
- Pydantic schemas
- async backend, если предыдущие этапы уже были async

---

# 7. Что не нужно вводить

На этом проекте **не нужно** использовать:
- full DDD
- CQRS
- Event Sourcing
- Mediator
- Unit of Work abstraction поверх SQLAlchemy
- repository interfaces/protocols для всего подряд
- generic base repository с множеством абстракций
- сложную permission framework
- микросервисы

Цель — сделать проект **реалистичным, но понятным**.

# 8. Этапы разработки

Ниже описано полное ТЗ проекта, чтобы была понятна общая картина.  
Однако реализация должна идти строго по **этапам**.

Нельзя переходить к следующему этапу, пока предыдущий:

- не завершён,
- не проверен,
- и не согласован.

После завершения каждого этапа нужно:

1. убедиться, что функционал работает;
2. привести код в аккуратный вид;
3. отправить этап на проверку;
4. только после проверки переходить к следующему этапу.

Если в ходе проверки будут замечания, этап считается незавершённым до их исправления.

## Этап 1. Базовый каркас проекта и auth

### Цель

Подготовить основу проекта и аутентификацию.
### Что нужно сделать

- создать проектную структуру по фичам
- настроить подключение к БД
- настроить Alembic
- настроить docker-compose
- реализовать модель пользователя
- реализовать JWT login
- реализовать endpoint `GET /auth/me`
### Сущность User

Минимальные поля:
- `id`
- `full_name`
- `email`
- `hashed_password`
- `role`
- `is_active`
- `created_at`
- `updated_at`

### Эндпоинты

- `POST /auth/login`
- `GET /auth/me` 

### Acceptance criteria

- пользователь может авторизоваться
- возвращается JWT token
- endpoint `/auth/me` возвращает текущего пользователя
- неавторизованный пользователь не получает доступ к защищённым endpoints

## Этап 2. Users management

### Цель

Сделать управление пользователями внутри системы.

### Что нужно сделать
- реализовать создание пользователя администратором
- реализовать список пользователей
- реализовать просмотр одного пользователя
- реализовать обновление пользователя
- реализовать деактивацию пользователя

### Бизнес-правила
- создавать пользователей может только `admin`
- новый пользователь не регистрируется сам
- email должен быть уникальным
- пароль должен храниться в хешированном виде
- Новым пользователям роли дает `admin` при создании 

### Эндпоинты

- `POST /users`
- `GET /users`
- `GET /users/{user_id}`
- `PATCH /users/{user_id}`
- `PATCH /users/{user_id}/deactivate`

### Acceptance criteria

- admin может создавать пользователя
- member не может создавать пользователя
- список пользователей доступен авторизованному пользователю
- данные пользователя корректно сохраняются в БД

## Этап 3. Clients module

### Цель

Реализовать модуль клиентов.

### Что нужно сделать

- миграция

### Дополнительно

- клиент появляется в списке
- данные клиента можно обновить
- `created_by` и `updated_by` корректно сохраняются

## Этап 5. Audit Logs

### Цель

Реализовать журнал действий пользователей.

### Что нужно сделать

- модель audit log
- запись важных действий в БД
- endpoint для просмотра логов

### Сущность AuditLog

Минимальные поля:

- `id`
- `actor_user_id`
- `action`
- `entity_type`
- `entity_id`
- `details`
- `created_at`

### Что логировать

Минимум:
- создание клиента
- обновление клиента
- создание задачи
- изменение статуса задачи
- создание пользователя
- изменение роли пользователя

### Эндпоинты

- `GET /audit-logs`

### Дополнительно

Можно добавить фильтры:

- по пользователю
- по типу сущности
- по действию
- по дате

### Бизнес-правила
- audit log создаётся при успешном выполнении действия
- audit log не должен подменять обычные технические логи приложения
- audit log используется как журнал действий в продукте

### Acceptance criteria
- важные действия сохраняются в БД
- в списке логов видно:
    - кто сделал действие
    - что сделал
    - над какой сущностью
    - когда сделал

# 9. Общие правила кода

## Naming

- понятные имена
- без лишних сокращений
- единый стиль по всему проекту

## Validation

- входные данные валидируются через Pydantic
- email должен быть нормализован
- enum-поля, статусы и формат телефона должны валидироваться и принимать только допустимые значения.
## Error handling

- ошибки должны быть понятными
- не должно быть голых `except`
- права доступа и not found кейсы должны обрабатываться корректно

## DB access

- вся работа с БД через repository
- service не должен писать SQL напрямую

## Audit

- важные изменения должны вызывать запись в audit log
